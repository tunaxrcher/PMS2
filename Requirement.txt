# Requirement ระบบบริหารจัดการรีสอร์ทและโรงแรม

## Resort / Hotel Property Management System (PMS)

---

# 1. ภาพรวมของระบบ

ระบบนี้เป็น Web Application สำหรับบริหารจัดการธุรกิจที่พัก เช่น รีสอร์ท โรงแรม วิลล่า หรือที่พักขนาดเล็กถึงกลาง โดยเน้นการใช้งานหลักของ Front Desk, Owner/Admin และ Housekeeping

เป้าหมายของระบบคือช่วยให้การทำงานประจำวันของที่พักเป็นระบบมากขึ้น ตั้งแต่การรับจองห้องพัก การจัดการข้อมูลลูกค้า การเช็คอิน เช็คเอาท์ การรับชำระเงิน การออกใบเสร็จ การจัดการสถานะห้อง การส่งงานให้แม่บ้าน การดูรายงาน และการตรวจสอบย้อนหลังผ่าน audit log

ระบบควรออกแบบให้เริ่มต้นใช้งานได้ง่ายในรูปแบบ MVP แต่โครงสร้างต้องรองรับการขยายในอนาคต เช่น รองรับโรงแรมหลายชั้น รีสอร์ทหลายโซน หลายสาขา การเชื่อมต่อ OTA เช่น Agoda / Booking.com การเชื่อมต่อเครื่องอ่านบัตรประชาชน เครื่องเขียนคีย์การ์ด ระบบ POS ระบบบัญชี และการแยกบิลหลายใบในอนาคต

แนวคิดหลักของระบบคือ

```txt
เริ่มจาก PMS สำหรับรีสอร์ทขนาดเล็กถึงกลาง
แต่โครงสร้างต้องสามารถขยายเป็น PMS สำหรับโรงแรมหรือหลายสาขาได้
โดยไม่ต้องรื้อฐานข้อมูลใหม่ทั้งหมด
```

---

# 2. เป้าหมายของระบบ

## 2.1 เป้าหมายระยะ MVP

ระบบเวอร์ชันแรกต้องสามารถทำงานหลักของธุรกิจที่พักได้ครบถ้วน โดยเน้นงานจริงที่ Front Desk, Admin และ Housekeeping ต้องใช้ทุกวัน

ฟีเจอร์หลักใน MVP ได้แก่

* Login / Logout
* จัดการผู้ใช้งานตาม role
* จัดการข้อมูลที่พัก
* จัดการโซนของที่พัก
* จัดการประเภทห้องพัก
* จัดการห้องพักจริง
* ดูสถานะห้องแบบปฏิทิน Room Grid
* สร้างการจอง
* แก้ไขการจอง
* บันทึกข้อมูลลูกค้า
* Assign ห้องจริงให้ booking
* ย้ายห้องให้ลูกค้า
* Check-in
* Check-out
* สร้าง Folio หลัก 1 ใบต่อ booking
* เพิ่มค่าใช้จ่ายเพิ่มเติม
* รับชำระเงินหลายช่องทาง
* บันทึกเงินมัดจำ
* ออกใบเสร็จเบื้องต้น
* เปลี่ยนสถานะห้อง
* สร้างงานทำความสะอาดให้แม่บ้าน
* แจ้งห้องเสียหรือ Out of Order
* ดูรายงานพื้นฐาน
* เก็บ audit log สำหรับรายการสำคัญ

---

## 2.2 เป้าหมายระยะยาว

ระบบต้องสามารถขยายไปสู่โรงแรมหรือหลายสาขาได้ โดยไม่ต้องรื้อโครงสร้างหลักใหม่

สิ่งที่ควรเผื่อไว้ตั้งแต่แรก ได้แก่

* Multi-property / Multi-branch
* Zone / Building / Floor / Villa Area
* แยก Room Type ออกจาก Physical Room
* แยก Inventory ออกจากห้องจริง
* รองรับ Rate Plan
* รองรับ Daily Rate
* รองรับ Package สำหรับรีสอร์ท
* รองรับ Split Folio
* รองรับ Channel Manager Integration
* รองรับ Smart Card Reader
* รองรับ Keycard Encoder
* รองรับ POS Integration
* รองรับ Accounting Integration
* รองรับ Mobile App หรือ Tablet App ในอนาคต
* รองรับ WebSocket สำหรับ real-time update
* รองรับรายงานขั้นสูง เช่น ADR, RevPAR, Forecast

---

# 3. ข้อกำหนดทางเทคนิคและสถาปัตยกรรม

## 3.1 Platform

ระบบเป็น Web Application ใช้งานผ่าน Browser เป็นหลัก

อุปกรณ์ที่ต้องรองรับ

* Desktop สำหรับ Front Desk และ Admin
* Tablet สำหรับ Front Desk หรือ Manager
* Mobile / Tablet สำหรับ Housekeeping

ระบบควรออกแบบแบบ Responsive Design โดยใน MVP ให้เน้น Desktop และ Tablet เป็นหลัก เพราะเป็นอุปกรณ์ที่เหมาะกับงาน Front Desk และ Dashboard มากที่สุด

---

## 3.2 API-first Design

ระบบต้องออกแบบ backend เป็น RESTful API เพื่อแยกระบบประมวลผลออกจากส่วนแสดงผล

เหตุผลที่ต้องใช้ API-first

* รองรับการสร้าง Mobile App ในอนาคต
* รองรับ Tablet App สำหรับแม่บ้าน
* รองรับการเชื่อมต่อ OTA
* รองรับการเชื่อมต่อ Smart Card Reader
* รองรับการเชื่อมต่อ Keycard Encoder
* รองรับการเชื่อม POS หรือระบบบัญชีภายนอก
* ทำให้ Frontend และ Backend พัฒนาแยกกันได้
* ทำให้ระบบสามารถ scale และ maintain ได้ง่ายกว่า

โครงสร้างภาพรวม

```txt
Frontend Web / Tablet / Mobile
        ↓
REST API Backend
        ↓
Business Logic Layer
        ↓
MySQL 8 Database
        ↓
Audit Log / Report / Integration
```

---

## 3.3 Database

ระบบจะใช้ฐานข้อมูลแบบ Relational Database โดยเลือกใช้ **MySQL 8 + InnoDB** เป็นฐานข้อมูลหลัก

เหตุผลที่ใช้ MySQL 8

* เหมาะกับระบบธุรกิจทั่วไปและระบบหลังบ้าน
* ทีม dev ส่วนใหญ่คุ้นเคย
* Hosting และ Cloud Provider รองรับกว้าง
* ใช้ร่วมกับ Prisma ได้ดี
* รองรับ transaction
* รองรับ row-level locking ผ่าน InnoDB
* รองรับ foreign key
* รองรับ index
* รองรับ JSON field
* เหมาะกับ MVP และสามารถขยายต่อได้

ข้อกำหนดสำคัญ

```txt
ต้องใช้ MySQL 8.x
ต้องใช้ Storage Engine เป็น InnoDB เท่านั้น
ห้ามใช้ MyISAM
```

เหตุผลที่ PMS ต้องใช้ Relational Database

* ต้องจัดการ transaction ทางการเงิน
* ต้องป้องกันการจองซ้ำ
* ต้องมีความสัมพันธ์ของข้อมูลชัดเจน
* ต้องทำรายงานรายได้
* ต้องทำรายงาน occupancy
* ต้องมี audit log
* ต้องรองรับ constraint
* ต้องรองรับ locking
* ต้องรองรับ data integrity

---

## 3.4 Security

ระบบต้องมีมาตรฐานความปลอดภัยเบื้องต้นดังนี้

* Password ต้องเก็บแบบ hashing เท่านั้น
* ห้ามเก็บ password เป็น plain text
* ต้องมีระบบ role-based access control
* ต้องจำกัดสิทธิ์การเข้าถึงตามหน้าที่
* ข้อมูลสำคัญ เช่น เลขบัตรประชาชนหรือพาสปอร์ต ต้องจำกัดสิทธิ์การมองเห็น
* ควร mask ข้อมูล sensitive บางส่วน เช่น แสดงเลขบัตรเฉพาะบางหลัก
* ทุก action สำคัญต้องถูกบันทึกใน audit log
* Backend ต้องตรวจ permission ทุกครั้ง ไม่พึ่งการซ่อนปุ่มที่ frontend อย่างเดียว
* ควรมี session expiration หรือ token expiration
* ควรมีการป้องกันการเข้าถึงข้อมูลข้าม property ในอนาคต

---

## 3.5 Audit Log

ระบบต้องบันทึก action log สำหรับรายการสำคัญทุกครั้ง เพื่อให้สามารถตรวจสอบย้อนหลังได้ว่าใครทำอะไร เมื่อไหร่ และเปลี่ยนข้อมูลอะไรไปบ้าง

ข้อมูลที่ควรเก็บใน audit log

* User ID
* Property ID
* Action
* Entity Type
* Entity ID
* Old Value
* New Value
* IP Address
* User Agent
* Timestamp

รายการที่ควรบันทึก log ได้แก่

* Login
* Logout
* สร้างการจอง
* แก้ไขการจอง
* ยกเลิกการจอง
* เปลี่ยนราคา
* เพิ่มส่วนลด
* รับชำระเงิน
* Void payment
* Refund
* Check-in
* Check-out
* ย้ายห้อง
* เปลี่ยนสถานะห้อง
* ตั้งห้องเป็น Out of Order
* แก้ไขข้อมูลลูกค้า
* สร้าง user
* เปลี่ยน role
* เปลี่ยน permission
* ลบหรือปิดใช้งาน user

โครงสร้างตารางตัวอย่าง

```txt
audit_logs
- id
- property_id
- user_id
- action
- entity_type
- entity_id
- old_value_json
- new_value_json
- ip_address
- user_agent
- created_at
```

หมายเหตุสำหรับ MySQL

* `old_value_json` และ `new_value_json` สามารถใช้ data type เป็น `JSON`
* ถ้าต้อง query audit log บ่อย ควรมี index ที่ `property_id`, `user_id`, `entity_type`, `entity_id`, `created_at`

# 4. User Roles และ Permission

## 4.1 Role ระดับ MVP

ใน MVP ระบบจะมีผู้ใช้งานหลัก 3 ระดับ

---

### 1. Admin / Owner

Admin / Owner เป็นผู้ใช้งานที่มีสิทธิ์สูงสุดของระบบ สามารถเข้าถึงทุกเมนูและจัดการข้อมูลหลักทั้งหมดได้

สิทธิ์หลัก

* เข้าถึงได้ทุกระบบ
* จัดการข้อมูลที่พัก
* เพิ่ม ลบ แก้ไขห้องพัก
* เพิ่ม ลบ แก้ไขประเภทห้อง
* จัดการโซน
* ตั้งราคา Base Rate
* จัดการ Rate Plan
* ดูรายงานรายได้
* ดูรายงาน occupancy
* ดูรายงาน booking source
* ยกเลิกการจอง
* คืนเงิน
* Void payment
* แก้ไขราคาพิเศษ
* อนุมัติส่วนลด
* สร้างบัญชีผู้ใช้งาน
* กำหนดสิทธิ์พนักงาน
* ดู audit log

---

### 2. Front Desk

Front Desk เป็นผู้ใช้งานหลักของระบบ ใช้สำหรับจัดการงานประจำวัน เช่น การจอง เช็คอิน เช็คเอาท์ และรับชำระเงิน

สิทธิ์หลัก

* ดู Room Grid
* สร้างการจอง
* แก้ไขการจอง
* บันทึกข้อมูลลูกค้า
* ค้นหาลูกค้าเก่า
* Assign ห้องพัก
* ย้ายห้องพักให้ลูกค้า
* Check-in
* Check-out
* รับชำระเงิน
* บันทึกเงินมัดจำ
* ออกใบเสร็จ
* เพิ่มค่าใช้จ่ายใน Folio
* เปลี่ยนสถานะห้องบางประเภท
* ตั้งห้องเป็น Out of Order ชั่วคราว
* แก้ไขข้อมูลลูกค้า

ข้อควรระวัง

* การแก้ราคาไม่ควรเปิดเต็มทั้งหมด
* การลดราคาควรกำหนด limit
* การคืนเงินควรให้ Admin ทำเท่านั้น
* การ Void payment ควรจำกัดสิทธิ์
* การดูข้อมูล sensitive ของลูกค้าควรจำกัด

---

### 3. Housekeeping

Housekeeping เป็นผู้ใช้งานสำหรับแม่บ้านหรือทีมทำความสะอาด ใช้งานผ่านมือถือหรือแท็บเล็ตเป็นหลัก

สิทธิ์หลัก

* ดูรายการห้องที่ต้องทำความสะอาด
* ดูโซนของห้อง
* กดเริ่มทำความสะอาด
* กดทำความสะอาดเสร็จ
* เปลี่ยนสถานะห้องจาก Dirty เป็น Clean
* แจ้งปัญหาห้องเสีย
* ใส่หมายเหตุเกี่ยวกับห้อง

Housekeeping ไม่ควรเข้าถึง

* ข้อมูลการเงิน
* ข้อมูลบัตรประชาชนหรือพาสปอร์ตของลูกค้า
* รายงานรายได้
* การแก้ไข booking
* การรับชำระเงิน
* การคืนเงิน
* การจัดการ user
* การตั้งราคา

---

## 4.2 Permission-based Design

ถึงแม้ MVP จะมีเพียง 3 role แต่ backend ควรออกแบบแบบ permission-based ตั้งแต่แรก ไม่ควร hardcode ตาม role อย่างเดียว

ตัวอย่าง permission

```txt
booking.create
booking.update
booking.cancel
booking.check_in
booking.check_out
booking.move_room

guest.create
guest.update
guest.view_sensitive

payment.receive
payment.void
payment.refund

folio.add_charge
folio.add_discount
folio.close

room.update_status
room.set_out_of_order

housekeeping.view
housekeeping.update_task

maintenance.create
maintenance.resolve

report.view
user.manage
price.override
audit_log.view
```

ข้อดีของ permission-based design

* เพิ่ม role ใหม่ได้ง่าย
* จำกัดสิทธิ์ละเอียดได้
* รองรับโรงแรมที่มีหลายแผนกในอนาคต
* รองรับ owner แบบ read-only
* รองรับ manager ที่ดูรายงานได้แต่คืนเงินไม่ได้
* รองรับ accountant ที่ดูการเงินได้แต่แก้ booking ไม่ได้

โครงสร้างที่ควรมี

```txt
users
roles
permissions
user_roles
role_permissions
```

# 5. โครงสร้างรองรับรีสอร์ทและโรงแรม

## 5.1 Property Management

ระบบควรมีตาราง Property ตั้งแต่แรก แม้ตอน MVP จะมีที่พักเดียว

เหตุผลที่ต้องมี Property ตั้งแต่แรก

* รองรับหลายรีสอร์ท
* รองรับหลายโรงแรม
* รองรับหลายสาขา
* รองรับ owner ที่ดูหลาย property
* รองรับ report แยกตาม property
* ป้องกันการรื้อ database ในอนาคต

โครงสร้างตัวอย่าง

```txt
properties
- id
- name
- property_type: resort / hotel / villa / hostel
- address
- phone
- email
- timezone
- check_in_time
- check_out_time
- tax_setting
- active
- created_at
- updated_at
```

ทุกตารางสำคัญควรผูกกับ `property_id`

เช่น

* rooms
* room_types
* zones
* bookings
* guests
* folios
* payments
* users
* reports
* audit_logs

---

## 5.2 Zone Management

ระบบต้องรองรับพื้นที่หลายรูปแบบ ทั้งรีสอร์ทแนวราบและโรงแรมหลายชั้น

ตัวอย่างโซนของรีสอร์ท

* Beach Zone
* Pool Villa Zone
* Garden Zone
* Riverside Zone
* Main Building
* Restaurant Area
* Spa Area

ตัวอย่างโซนของโรงแรม

* Building A
* Building B
* Floor 1
* Floor 2
* Wing A
* Wing B
* Lobby Zone

ควรออกแบบ zone เป็นโครงสร้างแบบ parent-child เพื่อให้ยืดหยุ่น

```txt
zones
- id
- property_id
- name
- zone_type: building / floor / wing / villa_zone / beach_zone / pool_zone
- parent_zone_id
- sort_order
- active
- created_at
- updated_at
```

ตัวอย่างโครงสร้างโรงแรม

```txt
Main Building
  └── Floor 1
      └── Room 101
      └── Room 102
  └── Floor 2
      └── Room 201
      └── Room 202
```

ตัวอย่างโครงสร้างรีสอร์ท

```txt
Beach Zone
  └── Villa A1
  └── Villa A2

Garden Zone
  └── Villa B1
  └── Villa B2
```

---

## 5.3 Room Type และ Physical Room

ระบบต้องแยก “ประเภทห้อง” ออกจาก “ห้องจริง”

### Room Type

Room Type คือประเภทของห้องที่ใช้ขายให้ลูกค้า เช่น

* Standard Room
* Deluxe Room
* Family Room
* Pool Villa
* Beachfront Villa
* Garden View Villa

โครงสร้างตัวอย่าง

```txt
room_types
- id
- property_id
- name
- description
- base_occupancy
- max_occupancy
- base_rate
- active
- created_at
- updated_at
```

### Physical Room

Physical Room คือห้องจริงที่มีอยู่ในที่พัก เช่น

* 101
* 102
* 201

# 6. Room Status และ Booking Status

## 6.1 Room Status

Room Status คือสถานะของห้องจริง ไม่ใช่สถานะของ booking

สถานะห้องระดับ MVP

```txt
Clean
Dirty
Occupied
Out of Order
```

สถานะที่ควรเผื่อในอนาคต

```txt
Vacant
Clean
Dirty
Occupied
Out of Order
Out of Service
Cleaning
Inspected
Reserved / Assigned
Due Out
Stay Over
```

ความหมายหลัก

```txt
Clean = ห้องสะอาด พร้อมขายหรือพร้อมให้ลูกค้าเข้าพัก
Dirty = ห้องต้องทำความสะอาด
Occupied = มีลูกค้าเข้าพักอยู่
Out of Order = ห้องเสีย ไม่พร้อมขาย
Out of Service = ห้องพักไม่เปิดขายชั่วคราว แต่ไม่ใช่ห้องเสีย
Cleaning = กำลังทำความสะอาด
Inspected = ทำความสะอาดเสร็จและตรวจแล้ว
Due Out = ลูกค้าจะ check-out วันนี้
Stay Over = ลูกค้าพักต่อ
```

---

## 6.2 Booking Status

Booking Status คือสถานะของการจอง

สถานะ booking ที่ควรมี

```txt
Pending
Confirmed
Checked In
Checked Out
Cancelled
No Show
```

ตัวอย่างการไหลของ booking

```txt
Pending → Confirmed → Checked In → Checked Out
Confirmed → Cancelled
Confirmed → No Show
```

เหตุผลที่ต้องแยก Room Status กับ Booking Status

* ห้องอาจ Dirty แต่ booking เป็น Checked Out
* ห้องอาจ Clean แต่ booking ยัง Confirmed
* Booking อาจยังไม่ assign ห้องจริง
* ห้องอาจ Out of Order โดยไม่มี booking
* Booking อาจถูก cancel แต่ห้องยัง Clean
* Booking อาจ No Show แต่ inventory ต้องถูกจัดการแยกต่างหาก

# 10. Check-out Flow

## 10.1 ขั้นตอน Check-out


* Partial payment

* Void payment
* Refund

* Check-out ไม่ได้ถ้ายังมี balance ค้าง

## 17.3 Overbooking Protection ด้วย MySQL

การป้องกัน overbooking ต้องทำที่ backend และ database ไม่ใช่ frontend อย่างเดียว

ใน MySQL ต้องใช้

```txt
MySQL 8
InnoDB
Transaction
Row-level Lock
SELECT ... FOR UPDATE
Unique / Foreign Key Constraint
```

ตัวอย่าง flow ตอน create booking

```txt
BEGIN TRANSACTION
1. Lock inventory rows ของ room_type และ date range ที่เกี่ยวข้อง
2. Check available_rooms
3. ถ้าห้องว่างพอ ให้ create booking
4. Create booking_rooms
5. Create folio
6. Update inventory
7. Create audit log
8. COMMIT
```

ถ้ามีพนักงาน 2 คนจองพร้อมกัน ระบบต้องรับประกันว่าไม่มีการขายเกินจำนวนห้องที่มี

ตัวอย่างแนวคิด SQL locking

```sql
SELECT *
FROM inventory
WHERE property_id = ?
AND room_type_id = ?
AND date BETWEEN ? AND ?
FOR UPDATE;
```

หลังจาก lock แล้วจึงตรวจสอบ `available_rooms`

ถ้าห้องว่างไม่พอ ต้อง `ROLLBACK`

```txt
ถ้า available_rooms < จำนวนห้องที่ต้องการ
→ ROLLBACK
→ แจ้ง user ว่าห้องไม่พอ
```

ข้อสำคัญ

* ห้ามเช็กห้องว่างเฉพาะที่ frontend
* ห้าม create booking โดยไม่ lock inventory
* ห้าม update inventory นอก transaction
* ต้องใช้ InnoDB เท่านั้น
* ควรมี index ที่ `property_id`, `room_type_id`, `date`

## 24.1 MySQL Database Design Notes

การออกแบบตารางใน MySQL ควรยึดหลักดังนี้

```txt
Storage Engine = InnoDB
Character Set = utf8mb4
Collation = utf8mb4_unicode_ci หรือ utf8mb4_0900_ai_ci
Primary Key = BIGINT หรือ UUID ตามแนวทางทีม
Foreign Key = ควรใช้กับ relation สำคัญ
Index = ต้องออกแบบตาม query จริง
JSON = ใช้กับ snapshot, payload, audit log
```

ตารางที่ควรมี index สำคัญ

```txt
bookings:
- property_id
- status
- check_in_date
- check_out_date
- booking_source_id

booking_rooms:
- booking_id
- room_type_id
- room_id
- check_in_date
- check_out_date

inventory:
- property_id
- room_type_id
- date

payments:
- folio_id
- payment_method
- paid_at
- status

audit_logs:
- property_id
- user_id
- entity_type
- entity_id
- created_at
```

Unique constraint ที่ควรพิจารณา

```txt
inventory:
unique(property_id, room_type_id, date)

rooms:
unique(property_id, room_number)

ota_bookings:
unique(channel_id, external_booking_id)
```

# 29. สรุปความสำคัญของระบบ

12. `packages` และ `booking_packages` เพื่อรองรับแพ็กเกจรีสอร์ท
13. `audit_logs` เพื่อความปลอดภัยและตรวจสอบย้อนหลัง
14. `MySQL transactions` เพื่อให้ workflow สำคัญปลอดภัย
15. `SELECT ... FOR UPDATE` เพื่อ lock inventory และป้องกัน overbooking

```txt
User drags booking

```

## 31.16 ORM

ระบบจะใช้ **Prisma ORM**

เหตุผลที่เลือก Prisma

* Type-safe
* ใช้งานกับ TypeScript ดี
* ทำงานกับ MySQL ได้ดี
* ทำ migration ได้
* Query อ่านง่าย
* เหมาะกับทีมที่ต้องการความเร็วในการพัฒนา
* ลด error จากการเขียน SQL ตรง ๆ
* เหมาะกับ CRUD และ business module ทั่วไป

ข้อควรระวัง

สำหรับ query ซับซ้อนมาก ๆ เช่น report, availability calculation หรือ inventory locking อาจต้องใช้ raw SQL บางส่วน

แนวทางที่แนะนำ

```txt
CRUD ทั่วไป = Prisma
Transaction สำคัญ = Prisma transaction
Inventory Locking = Prisma transaction + raw SQL SELECT ... FOR UPDATE ถ้าจำเป็น
Report หนัก ๆ = raw SQL / summary table ในอนาคต
```

## 31.17 Database

เหตุผล

* ทีม dev คุ้นเคย
* Hosting รองรับง่าย
* รองรับ transaction
* รองรับ row-level locking
* รองรับ foreign key
* รองรับ index
* รองรับ JSON
* ใช้กับ Prisma ได้ดี
* เพียงพอสำหรับ PMS MVP และขยายต่อได้

ระบบ PMS ต้องพึ่งพา transaction อย่างมาก โดยเฉพาะ

* Create Booking
* Check-in
* Check-out
* Payment
* Refund
* Move Room
* Cancel Booking
* Inventory Update

ข้อกำหนดสำคัญ

```txt
MySQL Version = 8.x
Storage Engine = InnoDB
Character Set = utf8mb4
ต้องมี index สำหรับ query สำคัญ
ต้องใช้ transaction ใน workflow สำคัญ
ต้องใช้ row lock เพื่อป้องกัน overbooking
```

## 31.23 Validation Strategy

ระบบควร validate 2 ชั้น

ใช้ React Hook Form + Zod

* date format
* phone format

* permission
* room availability
* booking status
* folio balance
* payment amount
* role access
* overbooking
* duplicate booking
* room status
* refund amount
* cancellation rule

หลักการสำคัญ

```txt
Frontend validation = ช่วยให้ใช้งานง่าย
Backend validation = เป็นตัวตัดสินสุดท้าย
```
